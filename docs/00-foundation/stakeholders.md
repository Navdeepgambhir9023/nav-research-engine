# Stakeholders

**Document ID**: 00.4
**Domain**: Foundation
**Status**: Draft

---

## Purpose

Maps who uses the ROS and how. Ensures that different user needs are served by the appropriate interfaces and documentation.

## Audience

- Product managers defining priorities
- Architects designing interfaces
- Contributors understanding user needs

## Stakeholder Map

### Primary Stakeholders

#### 1. Lead Researcher / Product Owner

**Role**: Guides the research direction and evaluates outputs
**Needs**:
- High-quality intelligence reports with clear evidence chains
- Visibility into system state and confidence levels
- Ability to direct research priorities
- Audit trail of decisions and reasoning
**Interfaces**: Weekly reports, mission dashboards, approval workflows

#### 2. Operations Team

**Role**: Runs the daily and weekly cycles
**Needs**:
- Clear runbooks with step-by-step procedures
- Clear failure recovery procedures
- Visibility into cycle status and anomalies
- Ability to escalate to human review
**Interfaces**: CLI tools, status dashboards, alerting

#### 3. Engineers / Contributors

**Role**: Builds and maintains the ROS
**Needs**:
- Clear architectural documentation
- Well-defined component contracts
- Testing frameworks for components
- Ability to deploy changes safely
**Interfaces**: Code, ADRs, technical documentation

---

### Secondary Stakeholders

#### 4. Analysts / Researchers

**Role**: Reviews and validates research outputs
**Needs**:
- Access to raw research materials
- Ability to annotate and comment on findings
- Tools for exploring the knowledge graph
- Export capabilities for external use
**Interfaces**: Query tools, visualization dashboards

#### 5. Leadership / Decision-Makers

**Role**: Consumes intelligence reports for strategic decisions
**Needs**:
- Executive summaries of key findings
- Prioritized opportunity lists
- Risk assessments with confidence levels
- Trend analysis and predictions
**Interfaces**: Weekly/monthly reports, dashboards

---

### External Stakeholders

#### 6. External Data Providers

**Role**: Provides data feeds (APIs, datasets) consumed by the ROS
**Needs**:
- Clear API contracts (see `06-integration/`)
- Rate limits and usage expectations
- Error reporting and feedback
**Interfaces**: API integrations, data contracts

#### 7. LLM Providers (Gemini, etc.)

**Role**: Provides AI capabilities for research processing
**Needs**:
- Well-structured prompts generated by the system
- Clear input/output contracts
- Token budget management
**Interfaces**: Integration contracts (see `06-integration/gemini-contract.md`)

---

### Implicit Stakeholders

#### 8. Future Contributors

**Role**: Will join the project and need to understand it quickly
**Needs**:
- Clear onboarding documentation
- Understanding of vision and principles
- Ability to find relevant documentation
- Path from newcomer to contributor
**Interfaces**: README, onboarding guide, glossary

#### 9. The Knowledge Base Itself

**Role**: The repository as a stakeholder
**Needs**:
- Quality gates that protect integrity
- Version control that preserves history
- Clear provenance for all knowledge
- Regular maintenance and grooming
**Interfaces**: Automated quality checks, curation workflows

## Dependency Analysis

The stakeholder map informs:

- **Documentation priorities**: Engineers and Operations have the highest priority
- **Interface design**: Different stakeholders need different views into the system
- **Quality gates**: Different thresholds for different consumers
- **Communication cadence**: Weekly for leadership, daily for operations
